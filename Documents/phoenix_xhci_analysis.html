1 What the Boot Log is Actually Showing

The boot reaches USB init successfully — PCIe link comes up, VL805 ID reads correctly, HCRST completes, all DMA ring addresses are set up properly. The failure happens during the Enable Slot command after port reset. Here is the key sequence from the log:

[xHCI] Controller RUNNING at 10us USBSTS=0x00000000 [xHCI] Controller RUNNING (fast) USBSTS=0x00000005 [xHCI] Post-RS=1 MCU HSE cleared USBSTS=0x00000001 ... [xHCI] Pre-scan USBSTS=0x0000001d ERDP_LO=0xc80b5c08 IMAN=0x00000003 [xHCI] WARNING: halted before port scan — recovery [xHCI] Recovery done USBSTS=0x00000005 [xHCI] Port 2: CONNECTED Super 5 Gbps PORTSC=0x00281203 [xHCI] Port 2 reset complete PORTSC=0x000202f1 ... (20,000 EHB poll iterations over ~2 seconds) ... [xHCI] poll: stolen ERSTBA=0x00000000_c80b6000 CRCR_LO=0x00000000 ← CRCR wiped! [xHCI] poll: USBSTS=0x00000005 evt_dequeue=0 evt_cycle=1 [xHCI] poll: evt_ring[0..3] = all zeros ← controller never processed TRB [xHCI] Enable Slot: event timeout [USB] Port 2: Enable Slot failed

The line CRCR_LO=0x00000000 at the first re-burst dump is the smoking gun. The VL805 MCU has overwritten the Command Ring Control Register to zero. Our Enable Slot TRB is sitting correctly in memory (cmd_ring[0] = ... 00002401 — type 9 = Enable Slot, cycle=1), but the controller has no idea where to find it because CRCR was cleared.

Root cause in one sentence: The VL805 has an internal MCU that runs its own xHCI init firmware. Every time we write RS=1, the MCU interprets it as a trigger to re-run its own initialization, overwriting CRCR (and potentially DCBAAP/ERSTBA/ERDP) with pointers to its own SRAM-based rings. Our Enable Slot TRB is invisible to the controller because CRCR has been redirected to SRAM.

2 Issue 1 — CRCR Split-Write Race Condition

Critical
CRCR, ERSTBA, and ERDP written as two separate 32-bit stores in all re-burst loops
drivers/usb/usb_xhci.c — run_controller() step 3b (lines ~880–891), retry loop (lines ~924–935), evt_ring_poll re-burst (~1403–1414), cmd_ring_submit recovery (~1205–1215)

The file's own comment on reg_write64() correctly explains the problem:

"This matters critically for CRCR: the VL805 latches the command ring pointer only when it sees the full 64-bit value in one transaction. Two sequential 32-bit writes cause the VL805 to silently discard CRCR_LO, leaving CRCR=0."

Yet every single re-burst loop — including the critical one in step 3b and the retry loop — uses two separate writel() calls. The MCU window to steal CRCR is exactly the gap between writing CRCR_LO and CRCR_HI. On a PCIe bus this gap can be 50–200 ns — plenty of time for the MCU to read the half-valid register and cache it internally as zero.

/* CURRENT — BAD: split writes, MCU steals the gap */
writel(launch_crcr_lo,  op + OP_CRCR_LO);   // ← MCU can fire here
writel(launch_crcr_hi,  op + OP_CRCR_HI);   // ← too late, MCU already read LO

/* SAME PROBLEM in every re-burst in evt_ring_poll, cmd_ring_submit */
writel((uint32_t)(r_crcr & 0xFFFFFFFF), op + OP_CRCR_LO);
writel((uint32_t)(r_crcr >> 32),        op + OP_CRCR_HI);

✅ Fix — Use reg_write64 for CRCR, ERSTBA, and ERDP in every burst

The reg_write64() function already exists and does the right thing — a single 64-bit STR instruction. Replace every pair of CRCR writel calls with it:

/* FIXED: single 64-bit write — VL805 latches the full pointer atomically */
reg_write64(op,  OP_CRCR_LO,    crcr_val);   // one 8-byte PCIe write
reg_write64(ir0, IR_ERSTBA_LO,  erstba_dma); // ditto for ERSTBA
reg_write64(ir0, IR_ERDP_LO,    erdp_dma);   // ditto for ERDP

This needs to be applied in all four locations: setup_cmd_ring(), run_controller() step 3b, the retry loop inside run_controller(), evt_ring_poll(), and cmd_ring_submit() recovery.

3 Issue 2 — Doorbell Timing Gap After CRCR Write

Critical
Doorbell is rung too late — MCU overwrites CRCR in the gap between the write and the ring
drivers/usb/usb_xhci.c — cmd_ring_submit() (lines ~1246–1253), evt_ring_poll re-burst (~1416–1419)

Even after fixing the split-write issue, there is a second race window: between writing CRCR and ringing doorbell 0, the MCU has ~100–500 µs to steal CRCR back. In evt_ring_poll(), the re-burst writes CRCR and then calls debug_print() before ringing the doorbell. Each UART byte at 115200 baud takes ~86 µs. A single debug_print call with a format string can take 2–8 ms — completely ruining the timing.

/* CURRENT — evt_ring_poll re-burst */
writel(...crcr_lo..., op + OP_CRCR_LO);
writel(...crcr_hi..., op + OP_CRCR_HI);
writel(0x00000002U,   ir0 + IR_IMAN);
asm volatile("dsb sy; isb" ::: "memory");
debug_print("...re-burst all regs...");  // ← costs 1-8ms, MCU steals CRCR here
writel(0, xhci_ctrl.doorbell_regs);      // ← doorbell lands AFTER MCU overwrote CRCR

✅ Fix — Ring the doorbell in the same barrier burst as CRCR write

/* FIXED: doorbell immediately after CRCR, no prints in between */
reg_write64(op, OP_CRCR_LO, crcr_val);
writel(0, xhci_ctrl.doorbell_regs);          // ring NOW, before MCU can react
asm volatile("dsb sy" ::: "memory");
/* Print AFTER doorbell — diagnostic info is still useful */
debug_print("[xHCI] re-burst + doorbell (iter %d)\n", ms);

The same principle applies everywhere a doorbell is rung after a register burst. The CRCR write and doorbell ring should be separated by nothing but a DSB, not a UART print or a delay loop.

4 Issue 3 — MCU Ownership Battle: The Deeper Architectural Problem

Critical
The VL805 MCU continues re-initialising after RS=1 and after Enable Slot command submission
drivers/usb/usb_xhci.c — run_controller(), port_scan(), evt_ring_poll(); kernel/pci.c — overall firmware approach

The VL805 has an embedded Cortex-M class MCU running SPI-loaded firmware. This firmware:

  • Watches for RS=1 and immediately runs its own xHCI startup sequence
  • Sets HSE ~60 µs after RS=1 as a "I am starting" signal
  • Overwrites CRCR, DCBAAP, ERSTBA, ERDP with pointers to its own SRAM rings at address ~0xc80c5000
  • Re-halts the controller (HCH=1) within 10–60 µs if it decides its own init is in progress

The boot log shows the pre-scan halted state USBSTS=0x0000001d = HCH + HSE + EINT + PCD. The MCU halted the controller again between the time run_controller() returned and the time port_scan() checked. This is confirmed by:

Pre-scan USBSTS=0x0000001d ERDP_LO=0xc80b5c08 IMAN=0x00000003 USBSTS bits: bit0=HCH(halted) bit2=HSE bit3=EINT bit4=PCD — all four set

The current run_controller() approach is essentially correct — it detects this and has a recovery path — but the Enable Slot timeout is happening because the MCU steals CRCR during the 10ms delay between port reset and Enable Slot submission (delay_ms(10) in enumerate_device()).

🔧 Fix A — Tighten the Enable Slot sequence (recommended short-term)

The combination of fixes from Issues 1 and 2 should be sufficient IF also applied to the immediate pre-Enable-Slot window. Specifically: immediately before calling cmd_enable_slot(), perform a full re-burst using reg_write64 for CRCR, then ring the doorbell atomically:

/* In enumerate_device(), before cmd_enable_slot() */
/* 1. Re-win the controller one more time */
/* 2. Write Enable Slot TRB to cmd_ring[cmd_enqueue] */
/* 3. DSB to flush TRB to Device memory */
/* 4. reg_write64(op, OP_CRCR_LO, crcr_val) — 64-bit atomic */
/* 5. writel(0, doorbell_regs) — ring NOW in same burst */
/* 6. asm volatile("dsb sy" ::: "memory") */
/* 7. Poll evt_ring for CCE — MCU may fire HSE but won't halt us */

🔧 Fix B — Pre-stage TRB before HCRST, burst within CNR=0 window

A more robust approach: write the Enable Slot TRB into the command ring before issuing a targeted HCRST. Immediately after CNR=0 clears (no delay, tight spin), write CRCR as a 64-bit write and ring doorbell 0 in one back-to-back pair. The MCU typically takes ~500 µs to restart after HCRST; our burst completes in <5 µs.

/* Step 1: Write Enable Slot TRB to memory (MCU doesn't know about it yet) */
cmd_ring[0]  = 0;
cmd_ring[1]  = 0;
cmd_ring[2]  = 0;
cmd_ring[3]  = (TRB_TYPE_ENABLE_SLOT << 10) | cmd_cycle;
asm volatile("dsb sy" ::: "memory");

/* Step 2: HCRST — this resets MCU internal state */
writel(CMD_HCRST, op + OP_USBCMD);
asm volatile("dsb sy; isb" ::: "memory");

/* Step 3: Busy-wait CNR=0, NO delays, NO prints */
while (readl(op + OP_USBSTS) & STS_CNR) {}

/* Step 4: Blast DCBAAP, CONFIG, ERST, ERDP — then CRCR+DB atomically */
writel(dcbaa_lo, op + OP_DCBAAP_LO);
writel(dcbaa_hi, op + OP_DCBAAP_HI);
/* ... other regs ... */
reg_write64(op,  OP_CRCR_LO,  crcr_val);   // 64-bit write
writel(CMD_RS | CMD_INTE, op + OP_USBCMD); // RS=1
writel(0, doorbell_regs);                   // DB0 — immediately
asm volatile("dsb sy" ::: "memory");

🔧 Fix C — Firmware mailbox approach (alternative, longer-term)

The VL805 firmware loaded from SPI has a mailbox interface (the PERST# + firmware load sequence already uses tag 0x00030058). If Broadcom/VIA documentation for the firmware's USB Host API is available, it may be possible to submit enumeration commands directly to the MCU rather than fighting it. This avoids the ownership battle entirely. However, this interface is not publicly documented and would require reverse engineering or binary analysis of the VL805 SPI firmware image.

5 Issue 4 — IRQ System: GIC Not Configured

High
irq.c has a stub GIC — no vectors routed, no USB interrupt ever delivered
kernel/irq.c — irq_init(), irq_unmask(), irq_eoi(); kernel/exceptions.S

The entire IRQ system is effectively a no-op. irq_init() clears a software table but never touches the BCM2711's GIC-400 controller hardware. The TODO comments are clear:

/* TODO: Set up GIC (Generic Interrupt Controller) */
/* TODO: Unmask in GIC */
/* TODO: Mask in GIC */
/* TODO: Send End-of-Interrupt to GIC */

Consequences for the current xHCI work:

  • The CMD_INTE flag written to USBCMD has no effect — even if the VL805 asserts the interrupt line, the CPU never receives it.
  • The code already polls the event ring directly, so the USB currently works in polling mode — no GIC dependency for the current boot sequence.
  • Once the Enable Slot fix works and the kernel tries to use USB for I/O (keyboard, mass storage), interrupt-driven transfers will silently fail. All I/O would need to remain in a polling loop.

✅ Fix — BCM2711 GIC-400 initialisation (minimum viable)

BCM2711 GIC-400 base: 0xFF840000. Distributor offset: 0x1000. CPU Interface: 0x2000.

#define GIC_DIST_BASE  0xFF841000ULL
#define GIC_CPU_BASE   0xFF842000ULL

/* In irq_init(): */
// 1. GICD_CTLR = 1  (enable distributor)
// 2. GICC_PMR  = 0xFF  (all interrupt priorities unmasked)
// 3. GICC_CTLR = 1  (enable CPU interface)
// 4. For each SPI: GICD_ISENABLERn bit = 1  (enable specific IRQ)
// 5. Set exception vectors (EL1 IRQ vector → irq_dispatch)

/* BCM2711 USB IRQ: VL805 xHCI → GIC SPI 75 (check DTS) */

Note: the exceptions.S vector table must be verified to route the EL1 IRQ to the irq_dispatch() C handler. If the vector table currently has a stub or infinite loop for IRQ entries, no interrupts will work regardless of GIC state.

6 Issue 5 — USB Core Completely Stubbed Out

High
usb_core.c transfers all return -1; usb_core.c and usb_xhci.c are not connected
drivers/usb/usb_core.c, drivers/usb/usb_hid.c, drivers/usb/usb_mass_storage.c

The three main USB transfer functions in usb_core.c are hard-coded stubs that return -1:

int usb_control_transfer(...) { return -1; }   // TODO
int usb_bulk_transfer(...)    { return -1; }   // TODO
int usb_interrupt_transfer(...){ return -1; }  // TODO

Furthermore, usb_enumerate_device() accesses dev->num_interfaces and dev->interfaces[] without these ever being populated — the descriptor parsing that populates these fields is also a TODO comment.

The xhci_init() function in usb_xhci.c does perform its own inline enumeration (Get Device Descriptor, string descriptors) but the results are never stored in a usb_device_t struct, so class drivers (HID, mass storage) can never bind to them. There is also a disconnect: kernel/usb_storage.c exists but uses none of the functions in drivers/usb/usb_core.c.

✅ Fix — Wire usb_core.c to xhci via a callback/ops table

Define a usb_hc_ops_t struct (host controller operations) in usb.h and have xhci_init() register it. Then usb_control_transfer() calls through the ops table:

typedef struct {
    int (*control)(usb_device_t*, uint8_t rt, uint8_t req,
                   uint16_t val, uint16_t idx, void*, uint16_t, int);
    int (*bulk)   (usb_endpoint_t*, void*, size_t, int);
    int (*intr)   (usb_endpoint_t*, void*, size_t, int);
} usb_hc_ops_t;

static usb_hc_ops_t *g_hc_ops = NULL;

/* xhci registers itself: */
void xhci_register_hc_ops(usb_hc_ops_t *ops) { g_hc_ops = ops; }

7 Issue 6 — Pre-Scan Drain Loop Does Not Update ERDP Per Event

High
The post-run drain loop increments evt_dequeue but only writes ERDP once at the end
drivers/usb/usb_xhci.c — xhci_init() post-run drain block (lines ~1958–1978)

The pre-scan event drain loop (which runs after run_controller() returns) does:

while (event_valid) {
    evt_dequeue++;   // increments the pointer
    ...
}
/* write ERDP only once at the end */
reg_write64(ir0, IR_ERDP_LO, clean_erdp);

If the VL805 MCU is simultaneously writing events to the ring, the controller checks ERDP after each event write. If ERDP points to a slot already consumed (but ERDP not yet advanced), the controller may think the ring is full and stop posting events. The correct approach is to call evt_ring_consume() (which advances ERDP individually) or write ERDP after each consumed TRB during draining.

Additionally, this drain loop has no cycle-bit check wraparound — if more than 64 events are posted during firmware startup, evt_dequeue will go out-of-bounds into evt_cycle territory.

✅ Fix — Use evt_ring_consume() for each event in all drain loops

/* Replace the open-coded drain loop with: */
int drained = 0;
uint32_t tmp[4];
while (1) {
    uint32_t dw3 = evt_ring[evt_dequeue * 4 + 3];
    if ((dw3 & TRB_CYCLE) != evt_cycle) break;
    evt_ring_consume(ir0, tmp);   // advances dequeue AND writes ERDP
    drained++;
    if (drained > EVT_RING_TRBS) break;  // safety: avoid infinite loop
}

8 Issue 7 — MMC Driver: Byte-Wide Register Access on 16/8-bit Offsets

High
SDHCI_BLOCK_COUNT, CLOCK_CONTROL, TIMEOUT_CONTROL, SOFTWARE_RESET are 16/8-bit registers accessed via 32-bit read-modify-write
drivers/mmc/mmc.c — SDHCI register map definitions and access helpers

The SDHCI spec defines several sub-word registers:

RegisterOffsetSDHCI WidthCurrent Access
SDHCI_BLOCK_COUNT0x0616-bit32-bit read at 0x04
SDHCI_CLOCK_CONTROL0x2C16-bit32-bit, overlaps TIMEOUT(0x2E)
SDHCI_TIMEOUT_CONTROL0x2E8-bit32-bit, stomps SOFTWARE_RESET
SDHCI_SOFTWARE_RESET0x2F8-bit32-bit, covers TIMEOUT

Using 32-bit writes to 0x2C will modify TIMEOUT and RESET at the same time. On BCM2711 the arasan SDHCI is strict about register widths. The boot log showed the arasan correctly negotiating 50 MHz and reaching FAT-formatted boot partition — so the SD card is reading correctly — but incorrect TIMEOUT/RESET writes could cause spurious timeouts or silent resets during block I/O.

✅ Fix — Add byte and half-word accessors

static inline void sdhci_write16(uint32_t off, uint16_t v) {
    *(volatile uint16_t *)(SDHCI_BASE + off) = v;
}
static inline void sdhci_write8(uint32_t off, uint8_t v) {
    *(volatile uint8_t *)(SDHCI_BASE + off) = v;
}
/* Use sdhci_write16 for CLOCK_CONTROL (0x2C) */
/* Use sdhci_write8  for TIMEOUT_CONTROL (0x2E) and SOFTWARE_RESET (0x2F) */

9 Issue 8 — DMA Buffer Size vs Linker Section Reserved Size Mismatch

Medium
xhci_dma_buf requires 270,336 bytes but linker comment references 0x1400 (5120 bytes)
drivers/usb/usb_xhci.c — DMA_BUF_SIZE calculation; kernel/linker.ld

The code comment in usb_xhci.c states the buffer needs:

DMA_BUF_SIZE = DMA_SCRATCH_PAGES_OFF + MAX_SCRATCH_PAGES * 4096
             = 0x2000 + 64 * 4096
             = 0x2000 + 0x40000
             = 0x42000 = 270,336 bytes

The comment also says the linker script reserves it with '. += 0x1400'. If the linker section is only 0x1400 = 5,120 bytes, the scratchpad pages (VL805 needs 31 × 4096 = 126,976 bytes) overflow into arbitrary kernel memory. This would cause the VL805 to DMA into kernel code/data — likely causing the halted/HSE states.

✅ Fix — Verify and correct linker.ld .xhci_dma section

/* In linker.ld, the .xhci_dma section should be: */
.xhci_dma (NOLOAD) : ALIGN(4096) {
    __xhci_dma_start = .;
    . += 0x42000;    /* 270,336 bytes — matches DMA_BUF_SIZE */
    __xhci_dma_end = .;
} >ram

Also verify mmu.c maps all pages from __xhci_dma_start to __xhci_dma_end as Device nGnRnE. If only 0x1400 bytes are being remapped, scratchpad pages at the end of the buffer are marked Normal WB (cacheable), causing DMA coherency failures.

10 Issue 9 — SMP: CPUs 1–3 Running WFE Without Proper Initialisation

Medium
Scheduler shows 4 CPUs but only CPU 0 is fully initialised; CPUs 1–3 in undefined state
kernel/sched.c — sched_init_cpu(); kernel/boot.S

The boot log shows "CPUs detected: 4" and "Scheduler initialized for 4 CPUs". However, sched_init_cpu() is only called for CPU 0 in the boot sequence. CPUs 1–3 are presumably spinning on a WFE loop from boot.S but are not held in a safe power-down or parked state with a defined exception vector. The idle_task_fn uses WFE which will wake on any event (including spurious), and without proper MPIDR-based spin tables, the secondary CPUs may jump to undefined addresses.

✅ Fix — Use ARMv8 spin table or PSCI for secondary CPUs

Either: (a) in boot.S, park CPUs 1–3 in a WFI/SEV spin table waiting on a mailbox address, then bring them up sequentially once per-CPU stacks and MMU state are ready; or (b) use the Pi 4's PSCI mailbox at 0xd8 (spin table entry point) which the bootloader configures. The secondary CPU MMU/stack state must be initialised before sched_init_cpu(n) is called for n > 0.

11 Issue 10 — enqueue_task() Not Protected by Per-CPU Spinlock

Medium
enqueue_task() modifies the linked list without holding cpu_sched[n].lock
kernel/sched.c — enqueue_task(), dequeue_task()

Each cpu_sched_t has a spinlock_t lock, but enqueue_task() modifies runqueue_head, runqueue_tail, and the prev/next pointers without taking the lock. If two cores call enqueue_task() on the same CPU's runqueue simultaneously, the linked list can become corrupt.

✅ Fix — Wrap list mutations with spinlock

void enqueue_task(cpu_sched_t *sched, task_t *task) {
    unsigned long flags;
    spin_lock_irqsave(&sched->lock, &flags);
    /* ... list manipulation ... */
    spin_unlock_irqrestore(&sched->lock, flags);
}

Same applies to dequeue_task() and schedule() when it accesses runqueue_head.

13 Alternative Approaches to the MCU Race

If the targeted register fixes (Issues 1–3) don't fully resolve the Enable Slot timeout, here are three distinct architectural approaches in order of complexity:

Approach A — Bare-metal HCRST "steal" with pre-staged TRB (Recommended next step)

Issue a dedicated HCRST before each command submission. Write the command TRB to memory before the HCRST so it survives the reset. After CNR=0, write all registers (using reg_write64 for CRCR), set RS=1, and ring the doorbell in a back-to-back burst with zero gaps. The MCU's re-init window is ~500 µs; our burst is ~5 µs. Win ratio is ~100:1. This is the approach the existing code is attempting but is failing due to Issues 1 and 2.

Approach B — Polled mode with HCRST between every command

Accept that the MCU will re-own the controller between commands. For each command: do a mini HCRST cycle, blast registers, submit one TRB, poll for CCE, then halt and wait. This is slower (adds ~2ms per command) but deterministic. Works well for boot-time enumeration where throughput doesn't matter. Not suitable for runtime I/O.

Approach C — Interrupt-based handoff using VL805 mailbox

The VL805 firmware communicates status via a specific MMIO register (the "firmware ready" Command bit 20 polled in pci.c). Some VL805 firmware versions expose a minimal command interface via these mailbox registers. If the firmware can be convinced to enter "pass-through" mode (letting the host take direct xHCI ownership), the MCU race disappears entirely. This requires firmware binary analysis and is a medium-term research item, not a quick fix.

What Is Working Well: The PCIe bring-up, L1ss disable, ATU window, HARD_DEBUG fix, PERST# logic, HCRST sequence, DMA buffer layout, reg_write64 concept, scratchpad allocation, and the overall xHCI spec §4.2 sequence are all correctly implemented. The boot log shows CAPLENGTH, HCIVERSION, MaxSlots, MaxPorts all reading correctly — the controller is live and responding. The VL805 firmware is running. Port 2 detects a SuperSpeed device and port reset completes. The problem is only in the final ~5 µs window between CRCR write and doorbell ring.
