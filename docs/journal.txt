SESSION 1: 2026-03-10  v1-v17  HSE root cause + controller running
═══════════════════════════════════════════════════════════════

BREAKTHROUGH: Controller running as of v17.

ROOT CAUSE OF HSE (all versions before v17):
  After HCRST the VL805 MCU re-runs its firmware init and posts
  events to its own internal ring, setting USBSTS.EINT.
  Writing RS=1 while EINT=1 causes the MCU to deliver that
  pending event using its stale internal ring pointer (not our
  ERSTBA) -> DMA to unmapped address -> HSE in <1ms.
  Fix: poll until EINT+HSE both clear after HCRST, then write
  all ring registers in a tight burst and set RS=1 immediately.

Secondary bugs fixed v13-v16:
  v13  PAGESIZE register read and validated (0x00000001 = 4KB OK)
  v14  ERST address corruption — missing (uint64_t) cast on
       xhci_dma_buf+DMA_ERST_OFF gave high word 0x00000010
  v15  HCRST reinstated (incorrectly skipped in v12)
  v16  Port change bits cleared after HCRST. PORTSC dump added.
       Revealed ports 0,1,2 have connected devices.
  v17  THE FIX — EINT poll after HCRST + tight register burst
       + retry loop on HCH only (CMD_HSEE=0)

v17 boot log confirms:
  [xHCI] MCU settled after 1ms  USBSTS=0x00000001
  [xHCI] attempt 1: USBCMD=0x00000004 USBSTS=0x00000005 RUNNING
  [xHCI] Controller running. USBSTS=0x00000001


SESSION 2: 2026-03-12  v18-v35  Enumeration stack + CNR deadlock
═══════════════════════════════════════════════════════════════

GOAL: Graft friend's enumeration stack (Enable Slot, Address
Device, GET_DESCRIPTOR) onto the running controller, plus add
MSI interrupt support.

── pci.c fixes ──────────────────────────────────────────────

Fix 1 — RGR1 write ordering (root cause of intermittent PERST#)
  start4.elf leaves RGR1=0x00000003.
  Writing MISC_CTRL while RGR1=0x3 silently drops the write
  (SCB_ACCESS_EN=0), causing PERST# every boot.
  Fix: clear RGR1 bit 1 first, wait 1ms, then write MISC_CTRL.

Fix 2 — Mailbox GET_POWER_STATE guard
  Calling SET_POWER_STATE when USB is already ON causes the VC
  to reset the VL805 MCU mid-init.
  Fix: call mbox_get_power_state() first; skip all mailbox
  calls if USB already ON.

Fix 3 — Memory Space ordering
  VL805 Command already reads 0x10100146 after firmware load
  (firmware sets Memory Space + Bus Master itself).
  Our write is harmless but the ordering is critical — Memory
  Space must be enabled BEFORE the CNR poll or the MCU stalls.

Fix 4 — MSI setup: xhci_setup_msi() in pci.c
  Walks VL805 PCIe cap list to MSI cap (ID=0x05, offset 0x90).
  Programs MISC_MSI_BAR_CONFIG_LO/HI with PCIe target address
  (xhci_dma_phys() + 0x1000 + 0xC0000000).
  Sets MSI Enable bit in cap dword (bit 16).
  irq_set_handler(180, xhci_irq_handler, NULL).
  irq_unmask(180) -> GICD_ISENABLER[5] bit 20.
  Falls back to INTx (INTID 175) if MSI cap not found.

── usb_xhci.c enumeration code ─────────────────────────────

Added between v18-v30 (friend's stack grafted in):
  cmd_ring_submit()     — submit TRB to command ring + doorbell
  xhci_wait_event()     — spin on pending_event_ready flag
  xhci_irq_handler()    — drain event ring on MSI, set flag
  xhci_dma_phys()       — return DMA buf physical base for MSI
  cmd_enable_slot()     — Enable Slot command + wait event
  cmd_address_device()  — setup input context, Address Device
  do_get_descriptor()   — EP0 transfer ring, GET_DESCRIPTOR

Bug: TRB_TC on command TRBs
  cmd_ring[b+3] had TRB_TC set on every TRB.
  TRB_TC (Toggle Cycle) belongs ONLY on the Link TRB.
  Fix: normal TRBs -> (type << TRB_TYPE_SHIFT) | cmd_cycle
       Link TRB only -> add TRB_TC.

Bug: No CRCR re-write before doorbell
  VL805 MCU can zero CRCR between ring setup and doorbell.
  Fix: re-write CRCR (cmd_ring_dma | cycle | RCS) immediately
  before db[0]=0 with dsb sy; isb fence between the two.

Bug: IRQ handler missing cycle bit check
  Events consumed without checking ev_ptr[3] & 1 vs evt_cycle.
  Stale TRBs from previous ring cycle treated as valid.
  Fix: check (ev_ptr[3] & 1) != evt_cycle before consuming.
  Toggle evt_cycle when evt_dequeue wraps (prevents drift).

── The CNR deadlock saga (v31-v35) ──────────────────────────

SYMPTOM: xhci_init failed silently after MaxSlots/MaxPorts
printed, with no further output from do_reset().

The problem was a pre-HCRST CNR poll in do_reset().  On VL805
cold boot CNR=1 always before HCRST, so the poll spun 2000ms
then returned -1.  This led to a series of wrong fixes:

  v32: Removed pre-HCRST CNR poll. Added HCRST.
       Result: HCRST timed out (USBSTS=0x00000815 after 5s).

  v33: Switched to direct write CMD_HCRST (no RMW).
       Added two-stage poll: wait HCRST bit clear, then CNR.
       Result: HCRST bit never cleared (stuck for 500ms).

  v34: Kept direct write, added USBCMD readback diagnostic.
       Readback showed 0x00000002 (bit set correctly) but
       HCRST bit still never self-cleared after 500ms.

ROOT CAUSE (from earlier session transcript):
  HCRST on VL805 triggers a SECOND MCU firmware init cycle.
  After that second init the MCU treats our ring pointers as
  "foreign" (it didn't write them) and fires its watchdog:
  overwrites CRCR/DCBAA, asserts HCH+HSE ~60us after RS=1.
  This was actually the same root cause as the HSE in Session 1.
  Session 1's "fix" (EINT poll) worked by accident.

  Linux xhci_pci never hits this because it runs on a VL805
  that start4.elf already warmed up and parked cleanly.
  We power-cycle via mailbox, forcing a cold boot that Linux
  never sees.

CORRECT SEQUENCE FOR VL805 COLD BOOT:
  1. Mailbox power-on
  2. Enable Memory Space + Bus Master
  3. ioremap MMIO
  4. Poll CNR in pci.c (MCU cold-boot settle window)
  5. xhci_setup_msi()
  6. xhci_init() — NO HCRST, just clear stale status bits

BOARD-SPECIFIC DISCOVERY (boardrev d03114):
  CNR NEVER clears on this board during cold boot.
  USBSTS stays 0x00000805 (HCH+HSE+CNR) throughout.
  This is NORMAL. The MCU IS running.
  MMIO reads work, PAGESIZE=1, RS=1 succeeds without HSE.
  Always proceed past the CNR poll timeout — it is not an error.

KEY RULES FOR FUTURE DEVELOPERS
═══════════════════════════════════════════════════════════════

1. HCRST rule is FIRMWARE-DEPENDENT — read Rule 16 first.
   OLD FIRMWARE (CNR stuck, no XHCI-STOP in VC4 log):
     NEVER issue HCRST.  The MCU is mid-init; HCRST triggers
     a second init cycle.  The MCU fights your ring pointers
     and asserts HSE ~60us after RS=1.
   NEW FIRMWARE (CNR clears fast, XHCI-STOP visible in VC4 log):
     HCRST IS REQUIRED before programming rings.  The MCU
     configures its own internal ERST during firmware init.
     Without HCRST, RS=1 causes the MCU to DMA-write to that
     stale ERST (GPU-side memory) → UR/CA from RC → HSE.
     See Rule 16 for the full conditional HCRST sequence.

2. CNR behaviour is FIRMWARE-DEPENDENT (not purely hardware).
   OLD FIRMWARE (no XHCI-STOP): CNR stays 1 throughout cold
     boot on boardrev c03112/d03114.  USBSTS=0x00000805 is
     NORMAL.  Always proceed past the CNR poll.
   NEW FIRMWARE (XHCI-STOP + PCI0 reset): CNR clears in < 5
     ticks (< 5ms) after NOTIFY_XHCI_RESET.  Fast CNR clear
     is the signal to take the HCRST path (Rule 16).
   In both cases: do not return -1 if CNR is still set after
   the poll.  Do not treat CNR=1 as a fatal error.

3. Memory Space must be enabled BEFORE calling xhci_init.
   The MCU needs it to DMA to its own internal buffers.
   Correct sequence: power-on -> Memory Space -> ioremap
   -> CNR poll -> xhci_setup_msi -> xhci_init.

4. Do NOT read-modify-write USBCMD while CNR=1.
   USBCMD readback is unreliable during cold boot on VL805.
   If you ever need to write USBCMD, write the value directly.

5. TRB_TC (Toggle Cycle) belongs ONLY on the Link TRB.
   Never set it on normal command TRBs — it corrupts dword 3
   and the controller will misparse the TRB type.

6. Re-write CRCR immediately before ringing doorbell[0].
   The VL805 MCU can zero CRCR in the window between ring
   setup and the doorbell. Use dsb sy; isb between the CRCR
   write and db[0]=0.

7. Check cycle bit in xhci_irq_handler before consuming.
   (ev_ptr[3] & 1) must equal evt_cycle. Toggle evt_cycle
   when evt_dequeue wraps to avoid drift across ring passes.

8. RGR1 must be cleared before writing MISC_CTRL.
   start4.elf leaves RGR1=0x3. Write to MISC_CTRL with RGR1
   bit 1 set is silently dropped. Clear RGR1 bit 1 first.

MILESTONE — After 130+ debugging sessions, a USB device was
detected on Port 1 for the first time.

  [xHCI] Port 0x00000001: CONNECTED speed=0x3 PORTSC=0x40200ee1
  [xHCI] Port reset done. PORTSC=0x40200e03

Port 1 PORTSC=0x40200ee1 decoded:
  CCS=1  (device connected)
  PED=0  (not yet enabled — reset needed)
  Speed=3  (SS 5Gbps — DeskPi hub/SSD)
  DR=1 (bit 30 set — this is why port 1 was SKIPPED in v37)

WHY BIT 30 WAS SET:
  Our companion port guard (if portsc & (1U<<30)) return) was
  intended to skip EHCI/OHCI companion ports. On VL805 bit 30
  is set on the SS port (likely DR = Device Removable flag).
  It is NOT a companion port indicator. Removed in v38.

ENABLE SLOT TIMEOUT (v38):
  Port reset completed (PRC cleared, PORTSC=0x40200e03).
  Enable Slot TRB submitted. xhci_wait_event timed out after 5s.
  The completion event WAS on the event ring but xhci_wait_event
  only checked pending_event_ready (set by MSI handler).
  MSI delivery timing on the first command after RS=1 was
  unreliable — the IRQ fired but our handler may have missed
  or the event arrived before the handler was fully live.

v39 FIX — Direct event ring polling:
  xhci_wait_event now polls the event ring directly (evt_ring_poll)
  every 1ms in addition to checking pending_event_ready.
  MSI/IRQ path is now an optimisation — no longer required for
  correctness. evt_ring_poll checks the cycle bit, consumes the
  slot, advances ERDP, and clears IMAN.IP.

  Also fixed: PRC (Port Reset Change, bit 21) W1C clear after
  port reset completes, plus 20ms settle before Enable Slot.

EXPECTED v39 BOOT OUTPUT:
  [xHCI] Port 1: CONNECTED speed=3 SS PORTSC=0x40200ee1
  [xHCI] Port reset done. PORTSC=0x40200e03
  [xHCI] Enable Slot: cc=0x1 slot=0x1
  [xHCI] Address Device OK slot=0x1
  [xHCI] Device: VID=... PID=... class=0x8
  [xHCI] Interface 0 class=0x8 sub=0x6 proto=0x50
  [MSC] USB mass storage device probed OK

KEY RULES ADDED:
  9. Bit 30 of PORTSC is NOT a companion port flag on VL805.
     Do not skip ports based on bit 30. CCS=0 is sufficient.
  10. xhci_wait_event must poll the event ring directly.
      Never rely solely on MSI delivery for command completions.
      The first Enable Slot after RS=1 fires before MSI is stable.
  11. Clear PRC (bit 21) W1C after port reset completes.
      Leave it set and the MCU may treat the next command as stale.

SESSION 4: 2026-03-14  v40-v41  Three-fix bundle + diagnostics
═══════════════════════════════════════════════════════════════

CONTEXT: After v39 first-device milestone, tested on cold boot
from a different SD card writer (primary writer was corrupting
writes silently).  Boot log still showed HCRST-era failure
signature (CRCR zeroed after doorbell, Enable Slot TIMEOUT).
Investigation revealed: three compounding bugs, plus HCRST had
been quietly re-introduced in a local build between sessions.

── Bug 1: HCRST re-introduced ───────────────────────────────

Symptom in boot log (old 8:20 build):
  [xHCI] pre-db:  USBSTS=0x00000815 CNR=1! HCH=1!
  [xHCI] post-db: CRCR_LO=0x00000000
  [xHCI] Enable Slot TIMEOUT

Root cause: do_reset() had CMD_HCRST re-added with comment
"CNR never clears without it" — but CNR not clearing is NORMAL
(journal Rule 2).  HCRST re-triggers the MCU second init cycle
which zeros CRCR immediately after our doorbell.

Fix: removed HCRST entirely from do_reset().  Now only clears
stale W1C bits (HSE, EINT, PCD) and returns.  This is the
correct sequence documented in v35.

── Bug 2: CNR hard abort in pci.c ───────────────────────────

pci.c had: if (cnr_timeout) return; (hard abort from xhci_pcie_init)
This meant the kernel never reached xhci_init() on boards where
CNR stays set — which is this board (boardrev d03114 / c03112).

Fix: CNR poll is now warn-and-continue.  pci.c logs
"CNR still set after 2000ms — proceeding" and calls xhci_init()
regardless.  This matches journal Rule 2.

── Bug 3: Double mbox_power_on_usb ──────────────────────────

mbox_power_on_usb() was called in BOTH pci.c (correct) AND
vl805_init.c (wrong).  The second call caused the VideoCore
to issue an internal PERST# equivalent, resetting the VL805 MCU
mid-init — after we had already set up ring pointers.  This
manifested as CRCR being zeroed silently between ring setup
and the first doorbell.

Fix: mbox_power_on_usb() removed from vl805_init.c entirely.
vl805_init.c now only calls mbox_notify_xhci_reset().
On c03112 boards that call returns -1 (EEPROM board, mailbox
notify not needed) — pci.c ignores the return value.

── Friend's fix bundle (applied 2026-03-14) ─────────────────

Three fixes from external contributor confirmed on working
hardware, applied as-is:

Fix A — ERST explicit 32-bit DWord layout (setup_event_ring):
  Original used volatile uint64_t* with erst[1] = EVT_RING_TRBS
  which placed the segment size in the HIGH 32 bits of DWord 1-2
  rather than in DWord 2 (size field) of xHCI spec Table 6-4.
  Fix: explicit uint32_t writes:
    erst32[0] = evt_dma & 0xFFFFFFFF  // Base [31:0]
    erst32[1] = evt_dma >> 32          // Base [63:32]
    erst32[2] = EVT_RING_TRBS          // Segment size
    erst32[3] = 0                      // Reserved

Fix B — RS=1 single write (run_controller):
  Original had 20-attempt retry loop re-writing USBCMD between
  attempts (clearing and re-setting CMD_RS).  Each iteration
  issued new USBCMD writes while CNR=1, which the MCU may
  interpret as ring re-validation requests.
  Fix: single writel(CMD_RS | CMD_INTE), 10ms delay_ms, proceed
  regardless.  HCH=1 after RS=1 with CNR=1 is expected on this
  board — log warning but do not abort.

Fix C — CRCR re-write before doorbell (cmd_ring_submit):
  (Already present from v35 but confirmed by friend's diff.
   Kept as-is.)

── Diagnostic improvements (v41) ────────────────────────────

Added to help diagnose event ring dead-lock:

evt_ring_poll periodic diagnostic:
  Every 500 empty-ring polls, logs: deq position, want_cycle,
  slot[3] raw value, IMAN, ERDP_LO, USBSTS.  Rate-limited to
  avoid flooding UART; fires every ~500ms during 5s Enable Slot

xhci_wait_event timeout dump:
  On timeout, logs full state: deq, want_cycle, slot[0-3],

  vs cycle bit inversion vs MCU not running.

ERDP before/after logging in evt_ring_poll:
  Logs ERDP_LO and IMAN before and after advancing the dequeue

KEY RULES ADDED:
  12. Never call mbox_power_on_usb() more than once per boot.
      The second call resets the MCU mid-init.  Check the full
      call path — it must appear in pci.c ONLY.
  13. ERST layout must use explicit uint32_t writes (DWords 0-3).
      Do not use uint64_t pointer arithmetic for ERST entries.
      The VL805 DMA engine reads the ERST as 32-bit DWords.
  14. RS=1 is a single write.  Never loop re-writing USBCMD.
      CNR=1 / HCH=1 after RS=1 is normal on this hardware.
  15. ALL 64-bit xHCI register writes (ERSTBA, ERDP, DCBAAP, CRCR)
      MUST use two separate 32-bit writel() calls — LO first, then HI.
      xHCI spec §4.20.3: a single 64-bit MMIO store generates one
      8-byte MWr TLP; VL805 may latch only the first DWORD and leave
      the upper half undefined.  reg_write64() in usb_xhci.c enforces
      this — do NOT replace it with a volatile uint64_t* store.

  16. CONDITIONAL HCRST: issue when and only when CNR cleared fast.
      Detection: after mbox_power_on_usb + NOTIFY_XHCI_RESET, poll
      USBSTS.CNR.  If it clears within the timeout → new firmware path.
      If it never clears → old firmware path (see Rule 1).
      New firmware path sequence in pci.c (BEFORE xhci_setup_msi):
        a. writel(CMD_HCRST, op + USBCMD)
        b. poll USBCMD.HCRST bit clear (up to 500ms, every 5ms)
        c. poll USBSTS.CNR=0 again (up to 2s, every 5ms)
        d. proceed to xhci_setup_msi() + xhci_init() as normal
      xhci_init/do_reset must NOT issue HCRST — pci.c owns this.
      Expected boot output (new firmware + HCRST path):
        [VL805] New firmware: issuing HCRST to clear MCU init state...
        [VL805] HCRST complete after Xms
        [VL805] Post-HCRST MCU ready (Xms)  USBSTS=0x00000001
      Followed by normal ring setup and RS=1 with no HSE.

  17. RS=1 HSE RETRY for new firmware path only (usb_xhci.c):
      After writing RS=1, if USBSTS.HSE fires and HCH stays set,
      AND pre-RS=1 USBSTS had CNR=0 (new firmware path):
        a. Clear HSE+EINT+PCD (W1C write to USBSTS)
        b. Wait 2ms
        c. Re-write CMD_RS | CMD_INTE to USBCMD
        d. Continue polling — repeat up to 8 times
      After the retry, the MCU is past its startup event post →
      no further HSE → HCH clears → controller running.
      On OLD firmware (CNR=1 in pre-RS USBSTS): this block is
      never entered.  Do NOT retry RS=1 on old firmware — that
      triggers a second MCU init cycle (Rules 1 and 14).

  ERSTBA can update the cache.

*** HISTORIC MILESTONE ***
After 143 boots, the VL805 xHCI event ring is finally working.
No-op CCE received. MCU DMA write-back confirmed. Enable Slot
returned slot=1. Address Device completed CC=0x01 (success).
MFINDEX started after first port reset (t0=0x062a running).

── HOW IT WAS SOLVED ──────────────────────────────────────────

Circle sample/08-usbkeyboard was built on the Linux build machine
(aarch64-linux-gnu- toolchain, STDLIB_SUPPORT=0). The link step
failed due to GCC 14 libgcc __getauxval issue, but the COMPILE
step succeeded — producing .o object files for Circle's xHCI
driver (xhcieventmanager.o, xhcidevice.o etc.).

Rob analysed the compiled Circle .o files directly, reading the
machine code / disassembly to understand the exact register write
sequence Circle uses for VL805 init. This revealed three critical
differences from Phoenix's sequence:

FIX 1 — INTE before RS=1 (separate write):
  Circle writes CMD_INTE alone FIRST, then CMD_RS | CMD_INTE.
  Phoenix was writing CMD_RS | CMD_INTE in a single write.
  The VL805 MCU needs to see INTE set before the run bit to
  know where to deliver events.
  Code:
    writel(CMD_INTE, op + OP_USBCMD);          // INTE first
    dsb sy; isb
    writel(CMD_RS | CMD_INTE, op + OP_USBCMD); // then RS=1

FIX 2 — ERSTSZ=1 BEFORE ERSTBA (reversed segment init order):
  Circle and Linux both write ERSTSZ=1 first, then ERSTBA.
  Phoenix always wrote ERSTBA first, then ERSTSZ=1.
  The VL805 MCU latches the segment count at ERSTBA write time.
  With ERSTSZ=0 when ERSTBA arrives, the MCU treats it as
  "zero valid segments" and never consults the event ring.
  Code:
    writel(1U, ir0 + IR_ERSTSZ);               // count FIRST
    reg_write64(ir0, IR_ERSTBA_LO, erstba_dma); // then base

FIX 3 — ERDP written with EHB=1 on init:
  Circle always ORs 0x8 (EHB bit) into the first ERDP write.
  Phoenix was writing ERDP with EHB=0 throughout.
  The VL805 MCU uses an EHB=1 write as a "host ready"
  handshake signal before it starts posting events to the ring.
  Per xHCI spec EHB is W1C so writing EHB=1 when EHB=0 is
  harmless on compliant hardware.
  Code:
    reg_write64(ir0, IR_ERDP_LO, evt_dma | 0x8ULL); // EHB=1

── ADDITIONAL DISCOVERY ───────────────────────────────────────

MFINDEX starts after port reset, not after RS=1:
  Boot143 revealed MFINDEX is static at RS=1 but starts
  ticking after the first USB port reset in xhci_scan_ports().
  MFINDEX t0=0x062a t5ms=0x0652 (running) — confirmed after PR.
  This explains every "MFINDEX static" boot since boot116.
  The VL805 MCU ties SOF frame generation to PHY activity,
  not to RS=1. MFINDEX check moved to after port scan.

── NEW RULES ──────────────────────────────────────────────────

Rule 38: Write CMD_INTE alone BEFORE CMD_RS | CMD_INTE.
         Single combined write does not work on VL805.

Rule 39: Write ERSTSZ=1 BEFORE ERSTBA. The MCU latches segment
         count at ERSTBA write time. ERSTSZ=0 at that moment
         means "zero segments" — MCU ignores the ring forever.

Rule 40: Write ERDP with EHB=1 (OR 0x8) on init only.
         This is the MCU's "host ready" handshake. Without it
         the MCU will not begin posting events even if the ring

         MFINDEX after xhci_scan_ports(), not after TRUE RUNNING.

  default). Enable Slot for slots 2 and 3 returned CC=9 (No
  Slots Available) because the MCU had only enabled 1 slot.

── BOOT148 — ADDRESS DEVICE SLOT ID IN TRB DW3 ───────────────

  Problem: Address Device returned CC=11 ("Slot Not Enabled
  Error") for ALL slots including slot 1 after boot147 changes.

  Root cause: xHCI spec §6.4.3.4 — Address Device Command TRB
  DW3 bits[31:24] must contain the Slot ID. Our cmd_ring_submit
  built DW3 as (type<<10)|cycle with no slot_id field. The MCU
  received slot_id=0 in DW3, which is never enabled → CC=11.

  Fix: added dw3_extra parameter to cmd_ring_submit(). Address
  Device caller passes (uint32_t)slot_id<<24. All other command
  callers pass dw3_extra=0. Forward declaration and function
  definition updated to match.

  Also fixed: misleading "Address Device timeout" error message
  renamed to "Address Device failed" (it was a CC check failure,
  not a real timeout).

Rule 43: Address Device Command TRB DW3 bits[31:24] = Slot ID.
         Without this the MCU sees slot_id=0 (always invalid)
         and returns CC=11 "Slot Not Enabled Error". The xHCI
         spec §6.4.3.4 is explicit; the field must be set.

Rule 44: cmd_ring_submit must propagate caller-supplied DW3 bits.
         Use a dw3_extra parameter ORed into the final DW3 word
         so each command type can supply its own upper fields
         without needing a separate submission path.

── BOOT148 POSTMORTEM — CC=0x11 IS HEX, NOT DECIMAL ──────────

  Bootlog147 (from boot148 binary) showed:
    cc=0x00000011 — we logged "CC=17 decimal = Context State Error"

  Previous session had diagnosed CC=11 decimal (= Slot Not Enabled)
  and concluded the slot_id fix was the cure.  Bootlog147 proved:
  boot148 DID fix the slot_id in DW3, but CC=17 (Context State Error)
  remained — it was already present in boot143+ and the boot143 "OK"
  was a false positive (old drain loop checked type=0x22 instead of
  type=0x21, accepting a PSCE as a successful CCE).

  CC=17 = 0x11 hex = "Context State Error" — xHCI spec Table 130.
  CC=11 = 0x0b hex = "Slot Not Enabled Error".
  These are completely different errors. Always read hex carefully.

── BOOT149 — READ HCCPARAMS1 CSZ BIT ─────────────────────────

  Root cause of CC=17: read_caps() read CAPLENGTH, HCSPARAMS1,
  HCSPARAMS2, RTSOFF, and DBOFF — but NEVER read HCCPARAMS1.
  The CSZ bit (HCCPARAMS1[2]) determines context entry size:
    CSZ=0: 32-byte entries  (our hardcoded assumption)
    CSZ=1: 64-byte entries  (possible VL805 setting)

  If VL805 has CSZ=1 and we write 32-byte-spaced contexts:
    ICC at offset 0   — correct (32B wide, MCU skips upper 32B)
    Slot Context at offset 32  — WRONG (should be 64)
    EP0 Context at offset 64   — WRONG (should be 128)
  The MCU reads garbage Slot/EP0 fields and returns CC=17.

  Fixes applied:
    read_caps(): add readl(base + CAP_HCCPARAMS1), extract
      AC64=(bit 0), CSZ=(bit 2), XECP=(bits[31:16]),
      store in xhci_ctrl, log HCCPARAMS1 value and context size.
    CTX_SIZE macro: changed from compile-time 32 to runtime
      (xhci_ctrl.csz ? 64U : 32U) so all context pointer
      arithmetic and dma_zero() calls use the correct stride.

  xhci_ctrl.csz is zero-initialised (static struct) so CTX_SIZE
  safely defaults to 32 if evaluated before read_caps().

Rule 45: ALWAYS read HCCPARAMS1 in read_caps(). The CSZ bit
         determines context entry size (32 vs 64 bytes). Blindly
         assuming CTX_SIZE=32 causes CC=17 on controllers with
         CSZ=1 because Slot/EP0 contexts land at the wrong offsets.

Rule 46: CC=0x11 (hex) = 17 decimal = "Context State Error".
         CC=11 decimal = 0x0B hex = "Slot Not Enabled Error".
         Print hex and verify which error code you actually have
         before diagnosing root cause.

── BOOT302 — INITIAL DRIVER ───────────────────────────────────

  Wrote drivers/net/genet.c from scratch: BCM2711 GENETv5 polling
  mode driver. TX/RX via Q16 (GENET_DMA_DEFAULT_QUEUE=16), 64
  descriptors per ring, 2048-byte buffers.  genet_init() resets
  UMAC, sets MAC address from mailbox, configures TX/RX DMA rings,
  autoneg-waits for PHY link (BCM54213PE), enables UMAC TX+RX.
  genet_send() / genet_poll_rx() implement single-frame TX/RX.
  No IRQs — polling only.  kernel/lib.c calls genet_init() from
  kernel_main() and polls for RX frames every 4 ms in wimp_task().

── BOOT314 — FIRST CONFIRMED FRAMES ON WIRE ───────────────────

  ARP and ICMP frames from the Pi (dc:a6:32:94:84:24) first
  appeared in Wireshark at boot314.  DHCP succeeded, gratuitous
  ARP sent, ICMP echo replies logged.  This boot is the critical
  reference: genet_apply_link() did NOT exist yet, so the OOB
  register retained the genet_init() value of ID_MODE_DISABLE=1.

── BOOT331 — ID_MODE_DISABLE REGRESSION ───────────────────────

  genet_apply_link() was added to set RGMII OOB control on link-up.
  A line was included: oob &= ~GENET_EXT_RGMII_OOB_ID_MODE_DISABLE.
  This cleared ID_MODE_DISABLE (bit 16), meaning the MAC now added
  its own TX clock delay on top of the PHY's internal delay.

  The regression was invisible at the time because bootlog349 showed
  RDMA pidx advancing (RX frames being received), which was taken as
  evidence that OOB was correct.  But RX and TX use separate clock
  paths: RDMA works on RX_CLK which the PHY also delays, and the MAC
  delay on RX_CLK is separate from TX_CLK.  TX was never verified in
  that boot.  No Wireshark capture was taken.

── BOOT337 — MDF_CTRL BUG ─────────────────────────────────────

  MDF_CTRL (MAC destination filter) was written with (1<<17)|(1<<16),
  enabling filter entries 16 and 17.  But the Pi MAC was programmed
  into entries 0 and 1.  Result: every received frame was silently
  dropped by the destination filter.  Symptom: rx counter advanced
  but mib (TX consumed) stayed at 7 forever — ARP replies and ICMP
  replies were never sent because no requests were received.
  Fix: MDF_CTRL=0 (disable filter entirely), rely on PROMISC mode.

── BOOT338 — TX DEAD: ROOT CAUSE HUNT ─────────────────────────

  After boot337 MDF fix, RDMA was confirmed working (pidx advancing,
  ARP/ICMP requests received and logged).  But TDMA consumed
  descriptors while ZERO Pi frames appeared in Wireshark.  The Pi
  sent ARP, DHCP, and ICMP replies from its own perspective, but
  nothing reached the wire.

  Evidence that confused the diagnosis:
  - TDMA CONS_INDEX advanced → DMA layer was working.
  - DHCP DISCOVER was never seen by the DHCP server → TX dead from
    the very first frame.
  - ARP debug print showed SHA=dc:a6:32:00:00:00 and SPA=0.0.0.0 →
    initially suspected wrong data in frame, but this was actually
    an AArch64 ABI vararg overflow bug (see Rule 47).

── BOOT339 — ROOT CAUSE: RGMII DOUBLE CLOCK DELAY ─────────────

  Root cause identified: the boot331 regression — ID_MODE_DISABLE=0.

  BCM54213PE operates in RGMII-ID mode: the PHY adds an internal
  clock delay on both TX_CLK and RX_CLK.  The MAC must NOT add its
  own delay.  With ID_MODE_DISABLE=0, the MAC also delayed TX_CLK:

    PHY TX delay + MAC TX delay = RGMII TX timing violation
    → PHY discards every incoming TX frame silently
    → TDMA consumed descriptors, nothing left the RGMII pads.

  The fix: remove `oob &= ~GENET_EXT_RGMII_OOB_ID_MODE_DISABLE`
  from genet_apply_link().  Preserve the init-time value of 1.

  Additional fixes in boot339:
  - SYS_TBUF_FLUSH_CTRL (0x00c) cleared at init.  This register is
    separate from UMAC_TX_FLUSH (0xb34); Linux always clears it.
  - debug_print calls with >7 integer args split into multiple calls
    to avoid AArch64 stack-spill vararg corruption (see Rule 47).

  boot339 result: DHCP BOUND to 192.168.0.145, Pi frames visible in
  Wireshark (dc:a6:32:94:84:24), ARP correct, ICMP replies for seq
  25-59 logged.  ping: 21/78 packets received, 73.1% loss, 10-18s RTT.

── BOOT340 — DRAIN LOOP + UNICAST ARP ─────────────────────────

  Two remaining problems after boot339:

  1. 73% packet loss / multi-second RTT.
     Cause: single genet_poll_rx() call per 4ms wimp_task tick.
     Background broadcast traffic (IPv6 multicasts, STP, LLDP) queued
     ahead of ICMP requests.  Each queued frame cost 4ms to drain,
     so N queued frames → N×4ms latency before ICMP was seen.
     Fix: replace `if (flen = poll_rx()) {...}` with
          `while ((flen = poll_rx()) > 13) {...}` — drain the entire
          64-slot ring per tick.  Bounded loop, can't stall WIMP.

  2. "Host is down" errors, repeated ARP retries.
     Cause: ARP reply sent to dst = ff:ff:ff:ff:ff:ff (broadcast).
     macOS/BSD only updates its ARP cache from unicast ARP replies
     per RFC 826.  A broadcast reply is delivered but not cached →
     requester retries ARP → ping sees "Host is down."
     Fix: ARP reply dst set to requester's MAC (g_rx_frame[6..11]).

  boot340 result (bootlog360 + wire10.pcapng):
  OOB = 0x00f10050 (ID_MODE_DISABLE=1 confirmed).
  DHCP BOUND immediately.  ICMP seq 3–60 replied; gaps: seq 10 and
  37 only (2 isolated drops, ~3.4% loss).  stalls=0, rfcs=0.
  Sub-4ms RTT.  Confirmed: "the money shot."

  boot341: Fixed GENET RX stability — STATUS=0 (clear OWN bit in

  boot342: Added RDMA loopback self-test at init using

── RULES ADDED THIS CHAPTER ────────────────────────────────────

Rule 47: AArch64 ABI vararg limit — integer arguments 1-7 go in
         registers x1-x7; argument 8 and beyond spill to the stack.
         Bare-metal debug_print that uses va_arg without a proper
         stack frame cannot reliably read stack-spilled args — they
         read as zero.  Symptom: MAC printed as XX:XX:XX:00:00:00,
         IP as 0.0.0.0.  Fix: split any debug_print with >7 integer
         args into multiple calls of ≤7 args each.

Rule 48: RGMII-ID means the PHY adds clock delay — the MAC must not.
         Set ID_MODE_DISABLE=1 in EXT_RGMII_OOB_CTRL for BCM54213PE.
         If the MAC also adds delay (ID_MODE_DISABLE=0), TX_CLK
         arrives with double delay → PHY discards every TX frame
         silently → frames consumed by TDMA but nothing on wire.
         RDMA may still work (RX_CLK is separate path) which can
         mislead diagnosis into thinking OOB settings are correct.

Rule 49: Always verify TX with a wire capture (Wireshark/tcpdump)
         before diagnosing RX.  RDMA advancing does NOT imply TDMA
         is putting frames on wire.  These are independent paths
         that can fail independently.

Rule 50: MDF_CTRL entry N must match the descriptor slot N used for
         the MAC address.  Writing (1<<17)|(1<<16) enables entries
         16/17 while the address is in entries 0/1 → every received
         frame dropped silently.  Or set MDF_CTRL=0 and use PROMISC.

Rule 51: ARP replies must be unicast (RFC 826 §Packet Reception).
         BSD/macOS kernels do not update their ARP cache from
         broadcast ARP replies.  The reply is accepted at layer 2
         but the sender's hardware address is not stored → ARP
         retries repeat → ping sees "Host is down."

Rule 52: RX drain loop, not single-poll.  Calling genet_poll_rx()
         once per 4ms tick causes O(N×tick) latency when N frames
         queue ahead of the one you care about.  Drain the entire
         ring in a while() loop per tick; the ring is bounded (64
         slots) so the loop cannot stall the task scheduler.

── RULES ADDED THIS CHAPTER ────────────────────────────────────

Rule 53: Any blocking poll loop that waits for a packet must drive
         the GENET RX ring itself.  Calling a passive queue-check
         (udp_recvfrom, dhcp_poll) without also calling
         genet_poll_rx() → net_rx_frame() means frames sit in the
         DMA ring forever.  The correct pattern (from DHCP, adopted
         by DNS): while (!timeout) { genet_poll_rx(); net_rx_frame();
         queue_check(); }.

Rule 54: arp_resolve() is cache-only and will always miss for a
         gateway that was never replied to unicast.  DHCP broadcast
         does not populate the ARP cache for the router MAC.  Any
         code that needs to send a unicast IP frame to a previously-
         unseen host (DNS query to gateway, first TCP SYN, etc.) must
         use arp_resolve_blocking() — which sends a who-has request
         and polls until the reply fills the cache.

Rule 55: PROVIDE linker symbols resolve to the same address when no
         binary payload is embedded.  start == end → size == 0.
         Always guard module_load_from_memory() calls with a size
         check; do not pass a zero-length buffer to a loader that
         validates a minimum header size.

Rule 56: #define XHCI_VERBOSE 0 is the correct approach for
         diagnostic blocks that were essential during a debug arc
         but are now always-pass.  Deleting them loses the ability
         to re-enable them when a regression surfaces.  Wrapping
         them in #if XHCI_VERBOSE keeps the code, eliminates the
         log noise, and makes the flag easy to find.

Rule 57: Apply the VERBOSE flag pattern consistently across all
         drivers.  Each driver gets its own flag (#define GENET_VERBOSE,
         #define XHCI_VERBOSE, etc.) so individual subsystems can be
         re-enabled independently during debugging.  The flag belongs
         at the top of the .c file immediately after its last #include.
         Error paths, milestone prints, and version/capability lines
         are always-on; per-register readbacks, per-frame dumps, and
         state readbacks after each init step go behind the flag.

Rule 58: filecore_find_path is the correct abstraction for all disc
         access above the IDA/SIN layer.  Any code that needs a file
         or directory should call filecore_find_path(path, &ent) and
         then filecore_read_file(ent.sin, (uint32_t)ent.size).  Never
         call fc_ida_to_data_lba or g_fc_bdev->ops->read directly
         from new code — those are implementation details that belong
         inside filecore.c.

    const char *src = suggested_name ? suggested_name : "Unnamed";

    if (hdr->title_ptr >= 0x34u && hdr->title_ptr < size) {

Rule 61: FileCore is case-insensitive.  All directory name lookups

Rule 62: The authoritative RISC OS file identity is the filetype
         in the directory entry load_addr field, not the file
         content or header structure.  When bits [31:20] of
         load_addr = 0xFFF (datestamped file), filetype =
         (load_addr >> 8) & 0xFFF.  Module filetype = &FFA.

CMUNGE 0.84 — AARCH64 CODE GENERATION
──────────────────────────────────────────────────────────────
CMunge is the RISC OS module header generator.  Options:
  -64bit   generate AArch64 assembly glue instead of ARM32 AOF
  -tgcc    emit GAS syntax (.s) rather than ObjAsm

The generated assembly sets bit 30 of the init_entry field in
the module header:
  init_entry = (_CMUNGE_init - _CMUNGE_origin) | 0x40000000

Phoenix's module_load_from_memory() checks bit 30 of init_raw
to distinguish AArch64 (bit 30 SET) from ARM32 (bit 30 CLEAR).
CMunge 0.84 writes exactly this discriminator.  The generated
init veneer sequence:
  STP x29/x30  — save frame pointer / link register
  STR x12      — save workspace pointer
  STP x0/x1    — save original arguments
  MOV x0, x12  — first arg to clib initialise = workspace ptr
  BL  _clib_initialisemodule
  LDP x0/x2    — restore args 0 and 1
  LDR x1       — restore arg 2
  BL  module_init
  LDP x29/x30  — restore frame
  RET

For an embedded standalone module the C-library hooks are no-ops:
  _clib_initialisemodule(void *pwp) { return 0; }
  _clib_finalisemodule(void *pwp)   { return 0; }

RULES ADDED THIS SESSION
──────────────────────────────────────────────────────────────

Rule 66: CMunge 0.84 -64bit -tgcc generates GAS AArch64 assembly
         glue that sets bit 30 of init_entry
         (init_entry = offset | 0x40000000).  This is the correct
         discriminator Phoenix reads.  Phoenix does NOT need to
         inspect the module_flags word at +0x30 — bit 30 of
         init_entry is authoritative.

Rule 67: The module_flags field at +0x30 holds an OFFSET to a
         flags word (parallel to title_ptr), not the raw flags
         value.  Logging the field as a hex integer is harmless,
         but do not treat it as a bitmask directly.

Rule 68: Never place embedded binary data sections (.test_module,
         module images, font blobs) after .bss in the linker
         script.  If .bss contains a large statically-allocated
         heap, objcopy -O binary will fill the entire heap gap
         with zeros, producing an enormous flat binary.  Embedded
         read-only data must come before .bss.

Rule 69: Standalone AArch64 RISC OS modules for Phoenix must stub
         _clib_initialisemodule and _clib_finalisemodule to return
         0 (NULL error pointer).  These hooks are called by the
         CMunge -64bit veneer before and after the user init/final
         functions.  Without the stubs the link fails with
         undefined reference.

Rule 70: The CMunge -64bit CMHG syntax for Phoenix modules
         requires only: title-string, help-string,
         initialisation-code, finalisation-code.  The
         module-is-cpp field is not recognised by CMunge 0.84
         and causes a fatal "Unknown field" error; omit it.
